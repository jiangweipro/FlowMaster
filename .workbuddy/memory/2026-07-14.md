# 2026-07-14

## 修复执行阶段终端无输出
- 用户点击"执行 关闭"后，终端显示"运行中"但没有任何输出
- 原因分析：子进程可能启动失败或立即输出到 stderr，但 `TerminalBridge.attach` 是在 `spawnProcess` 之后调用的，导致 spawn 瞬间产生的错误/输出在监听器注册前丢失
- 修复 `src/terminalBridge.ts`：
  - `startProcess` 现在会输出工作目录和实际执行的命令行（便于排查命令是否被正确调用）
  - `attach` 之后立即 flush `processManager` 的 buffer，把 spawn 瞬间已经产生的输出补发到 WebView
- 更新 `tests/terminalBridge.test.ts`：用更稳定的查找方式定位 stdout/stderr 数据消息
## 修复审核按钮消失
- 用户反馈：切换到某个阶段后，"审核通过 / 打回" 按钮消失，状态显示"审核：unknown"
- 原因：审核按钮和状态一直基于 `current_phase`，而用户可能切换到其他阶段查看/操作，导致该阶段没有 gate 数据或状态不对
- 修复 `src/extension.ts`：
  - 审核按钮和状态改为基于当前选中的阶段 `selectedPhase`
  - 前端点击审核时带上 `phase` 字段，让后端知道要审核的是哪个阶段
  - 后端 `reviewGate` 优先使用前端传入的 `phase`，回退到 `current_phase`
## 将内置终端从 xterm.js 改为纯文本终端
- 用户多次反馈内置终端无输出，且 xterm.js 在 VS Code Webview 中加载/渲染不稳定
- 重构 `src/extension.ts`：
  - 移除 xterm.js 及其 addon 的 script/css 引用
  - 用 `<pre id="terminal-output">` 替代 xterm 渲染区域
  - 所有终端输出直接追加到 `<pre>` 元素并自动滚动
  - 保留标题栏、状态显示（就绪/运行中/已退出/错误）和 placeholder
- 优点：不再依赖外部脚本加载，输出始终可见，更稳定
- 编译、测试、打包通过，版本升到 0.2.5
- 已推送到 GitHub

## 修复左侧需求点击无法在主面板展示
- 用户反馈：左侧点击需求后，主面板没有显示对应需求详情
- 原因：主面板 WebView 没有 `ready` 握手，扩展在 WebView 还没准备好时就发送 `stateUpdated`，消息可能丢失；面板未就绪时从侧边栏选择需求也无法同步
- 修复 `src/extension.ts`：
  - 主面板 JS 加载完成后发送 `ready` 消息
  - 扩展收到 `ready` 后才调用 `sendSelectedDemand()` 下发数据
  - 新增 `panelReady` 和 `pendingDemandSelection` 标志，处理面板未就绪时的选择请求
  - 面板关闭时重置 `panelReady`

## 阶段卡片点击展示对应操作
- 用户反馈：点击阶段卡片后希望看到该阶段对应的具体操作

  - 关闭阶段（closure）隐藏执行按钮，显示“已完成”
